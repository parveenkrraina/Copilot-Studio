### Before You Start

1. **Check existing issues** to avoid duplicate work
2. **Open an issue** to discuss major changes before implementing
3. **Review existing labs** to understand the style and format
4. **Test your changes** thoroughly before submitting

### Lab Writing Standards

When creating or updating labs, follow these standards:

#### Structure
- **Clear objective**: State what learners will accomplish
- **Time estimate**: Realistic completion time
- **Prerequisites**: Required knowledge and resources
- **Step-by-step instructions**: Numbered, clear, actionable
- **Success criteria**: Checklist of learning outcomes
- **Troubleshooting section**: Common issues and solutions
- **Additional resources**: Links for deeper learning

#### Writing Style
- Use clear, concise language
- Write in second person ("you will...")
- Use active voice
- Include examples and scenarios
- Define technical terms on first use
- Use consistent terminology throughout

#### Formatting
- Use markdown formatting consistently
- Include emojis for section headers (ğŸ¯, â±ï¸, ğŸ“‹, etc.)
- Use code blocks for code snippets
- Use bold for UI elements (**"Create"** button)
- Use numbered lists for sequential steps
- Use bullet lists for non-sequential items

#### Technical Accuracy
- Test all instructions personally
- Verify all links work
- Ensure screenshots are current
- Confirm version compatibility
- Include specific version numbers when relevant

### Code and Sample Standards

If contributing code or samples:
- Follow Microsoft coding standards
- Include clear comments
- Provide usage examples
- Test thoroughly
- Document any dependencies

## ğŸ”„ Contribution Process

### 1. Fork the Repository
```bash
git clone https://github.com/parveenkrraina/Copilot-Studio.git
cd Copilot-Studio
```

### 2. Create a Branch
```bash
git checkout -b feature/your-contribution-name
```

Use branch naming conventions:
- `feature/lab-X-improvements` for lab enhancements
- `fix/lab-X-typo` for bug fixes
- `docs/update-readme` for documentation
- `resource/new-template` for resource additions

### 3. Make Your Changes
- Follow the guidelines above
- Keep changes focused and atomic
- Test your changes
- Update related documentation

### 4. Commit Your Changes
```bash
git add .
git commit -m "Clear, descriptive commit message"
```

Commit message format:
- Use present tense ("Add feature" not "Added feature")
- Be specific and descriptive
- Reference issue numbers if applicable (#123)

Examples:
- `Add Lab 6: Advanced Analytics and Reporting`
- `Fix typo in Lab 2 Step 5`
- `Update integration examples in Lab 4`
- `Add troubleshooting section to Lab 1`

### 5. Push to Your Fork
```bash
git push origin feature/your-contribution-name
```

### 6. Submit a Pull Request

1. Go to the original repository
2. Click "New Pull Request"
3. Select your fork and branch
4. Fill out the PR template:
   - **Title**: Clear, descriptive title
   - **Description**: What changes you made and why
   - **Related issues**: Link to related issues
   - **Testing**: How you tested the changes
   - **Screenshots**: If applicable

### 7. Review Process

- Maintainers will review your PR
- Address any feedback or requested changes
- Once approved, your contribution will be merged
- You'll be added to the contributors list!
